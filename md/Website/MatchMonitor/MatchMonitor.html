📋 Overview

Purpose The Match Monitor page displays dangerous matches and allows PIC to verify fraudulent transactions and mark customers as Problem Account (PA).

🎯 Key Features

  • Search matches by date/sport/bettype
  • View fraud ticket details
  • Verify suspicious transactions
  • Manual association detection
  • Mark customers as PA
  • Export analysis to Excel

👥 User Roles

  • Monitor Team: View & verify tickets
  • Host Team: Full control (Hold Key, Mark PA)

⚡ Data Freshness

  • Max 5-minute delay (job frequency)
  • Real-time customer CC status

📋 Data Flow Process (4 Steps)

Complete Flow Overview The following steps show how tickets are detected and customers are classified:

Step 1 Ticket Detection

Background jobs detect suspicious betting patterns and mark tickets as fraudulent:

  • Job: md/MatchMonitor/GroupBetting - General Group Betting Detection
  • Job: md/MatchMonitor/SabaGroupBetting - Saba Platform Group Betting Detection
  • Detection: Using 5 criteria (Device, AI, IP, 3 Matches, IP Last 3 Days) for general, 4 criteria for Saba
  • Output: Suspicious tickets identified and grouped by match

Step 2 Ticket Labeling

Detected tickets are labeled with fraud reason:

  • Table: CTSMatchMonitorDetail
  • Field: Reason = "Group Betting (GB)"
  • Data: TransID, CustID, MatchID, Refno, BetChoice
  • Purpose: Mark tickets as fraudulent for review

Step 3 General Customer Classification

Background job assigns Customer Class (CC) for customers with GB-labeled tickets:

  • Job: md/General/MatchMonitorClassification
  • Frequency: Every 5 minutes
  • Processing: One match at a time (FIFO)
  • Action: CustID with GB tickets → CategoryName = "System Detect Group Betting"
  • Table Updated: CTSCustomerClassification
  • Scope: All sports (cross-sport detection)

Step 4 Sport-Specific Customer Classification

Background job assigns sport-specific Customer Class for GB-labeled tickets:

  • Job: md/BySport/MatchMonitorClassificationBySport
  • Frequency: Every 5 minutes
  • Processing: Multiple matches per run (parallel)
  • Condition: IF match.SportType = "Saba Soccer"
  • Action: CustID with GB tickets → CategoryName = "Saba System Detect Group Betting"
  • Table Updated: CTSCustomerClassification
  • Scope: Sport-specific (Saba Soccer only for this case)
⚠️ Important Notes
  • Step 3 vs Step 4: Both jobs may classify the same customer, but Step 4 provides more specific sport-level classification
  • CategoryName Priority: Sport-specific categories (Step 4) typically take precedence over general categories (Step 3)
  • Data Freshness: Maximum 5-minute delay due to job frequency
  • Match Visibility: Match appears on this page after Step 2 completes (tickets labeled)
  • Customer CC Status: Updated after Step 3 or Step 4 completes

📝 Data Write Operations (Jobs → Database)

Background Jobs

🏅 MatchMonitorClassificationBySport

Sport-Specific

  • Frequency: Every 5 minutes
  • Purpose: Detect group betting by sport
  • Processing: Multiple matches per run
  • Reference: md/BySport/MatchMonitorClassificationBySport

🌐 MatchMonitorClassification (General)

Cross-Sport

  • Frequency: Every 5 minutes
  • Purpose: General match monitoring
  • Processing: One match at a time (FIFO)
  • Reference: md/General/MatchMonitorClassification

Tables Written by Jobs

Table Name Database Operation Description
CTSMatchMonitor CTS_DataCenter (MySQL) INSERT/UPDATE Match detection data (MatchID, SportType, Reason, EventDate)
CTSMatchMonitorDetail CTS_DataCenter (MySQL) INSERT Fraud ticket details (TransID, CustID, Refno, BetChoice)
CTSCustomerClassification CTS_DataCenter (MySQL) UPDATE Customer CC status (CategoryID, IsPA, CategoryName)
Classification Logic When detection rules are satisfied:
  • MatchID has multiple tickets
  • Job runs detection rules (group betting patterns)
  • CustID with fraud ticket → marked as CC (Problem Account)
  • Assign specific CC category (e.g., "Group Betting")

🔗 Data Dependencies Summary

Critical Dependencies Match Monitor page cannot function without:
  • Background Jobs: Must run to populate CTSMatchMonitor tables
  • CTS_DataCenter: Primary data source for match detection
  • MainDB: Master data for match info and transactions

📥 Data Inputs (Dependencies)

  • Background Job: MatchMonitorClassificationBySport
  • Background Job: MatchMonitorClassification (General)
  • Database: CTS_DataCenter (Match & Customer data)
  • Database: bodb_VR2Model (Match master & Transactions)

📤 Data Outputs (Actions)

  • Verify fraud transactions (update CTSMatchMonitorDetail)
  • Mark customers as PA (update CTSCustomerClassification)
  • Hold Key for matches (manual intervention)
  • Export analysis reports (Excel)
